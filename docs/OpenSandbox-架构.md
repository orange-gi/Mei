# OpenSandbox 架构文档

## 一、系统整体架构

OpenSandbox 采用分层架构设计，从上到下依次为 SDK 层、规范层、运行时层和沙箱实例层。这种分层设计遵循了关注点分离的原则，每一层都有明确的职责边界，层与层之间通过定义良好的接口进行通信。下面通过架构图展示各层之间的关系以及数据的流向。

从上述架构图可以看出，开发者通过 SDK 与系统进行交互，SDK 会根据操作类型将请求发送到相应的规范层。沙箱生命周期管理请求（如创建、删除、暂停沙箱）发送到生命周期规范对应的后端服务，而代码执行、文件操作等请求则直接发送到沙箱实例内部运行的 execd 守护进程。运行时层负责将请求路由到正确的沙箱实例，并管理沙箱的生命周期。

## 二、核心数据流图

### 2.1 沙箱创建流程

沙箱创建是 OpenSandbox 最核心的操作之一，理解其数据流对于掌握整个系统的工作原理至关重要。当用户调用 SDK 的「create」方法时，系统会执行一系列复杂的操作来创建并初始化一个可用的沙箱实例。

整个创建流程从用户发起创建请求开始，服务器首先拉取指定的容器镜像，然后注入 execd 守护进程二进制文件，接着创建容器并修改入口点以确保 execd 能够先于用户进程启动。容器启动后，execd 会启动内部的 Jupyter Server 用于代码解释功能，最后向服务器报告沙箱就绪状态，服务器将沙箱信息返回给用户。

### 2.2 代码执行流程

代码执行是 AI 应用最常用的功能，当用户在沙箱中执行代码时，数据流涉及多个组件的协作。以下流程图展示了从用户提交代码到获取执行结果的完整过程。

代码执行流程的关键在于 Jupyter 协议的集成。execd 守护进程内部维护与 Jupyter Server 的 WebSocket 连接，当用户提交代码时，execd 将请求转发给 Jupyter Server，再由 Jupyter Server 分发给对应的内核（Python、Java、JavaScript 等）执行。执行结果通过 WebSocket 逐层返回，最后由 execd 转换为 SSE（Server-Sent Events）事件流发送给 SDK。

### 2.3 文件操作流程

文件操作是沙箱与外部世界交换数据的主要方式，OpenSandbox 提供了完整的文件系统 API 来支持文件的上传、下载和管理。

文件操作请求通过 SDK 发送到 execd 守护进程的文件系统 API 端点，execd 接收请求后直接在容器文件系统上执行相应的操作。写入操作会将数据持久化到容器的文件系统中，读取操作则从容器文件系统获取数据并返回给客户端。

## 三、核心模块关系图

### 3.1 SDK 模块结构

OpenSandbox 提供的 SDK 封装了所有与沙箱交互的功能，主要包括 Sandbox 类、Filesystem 类、Commands 类和 CodeInterpreter 类。这些类之间的关系以及它们提供的核心方法如下所示。

Sandbox 类是整个 SDK 的核心入口，它聚合了 Filesystem、Commands 和 CodeInterpreter 三个组件。用户首先创建 Sandbox 实例，然后通过该实例访问其他组件的功能。每个组件都提供了丰富的异步方法，支持高效的并发操作。

### 3.2 服务器端模块结构

服务器端采用 FastAPI 框架构建，主要负责沙箱的生命周期管理。以下模块图展示了服务器端的核心组件及其职责。

LifecycleManager 是服务器端的核心管理器，负责协调其他所有组件。DockerRuntime 和 K8sRuntime 是两种不同的运行时实现，它们实现了统一的接口，LifecycleManager 可以根据配置选择使用哪种运行时。Router 负责将外部请求路由到正确的沙箱实例，AuthManager 则处理身份验证和授权。

### 3.3 execd 守护进程模块结构

execd 是注入到每个沙箱实例中的核心组件，它负责在沙箱内部执行代码、运行命令和管理文件。以下模块图展示了 execd 的内部结构。

WebServer 是 execd 的 HTTP 服务器，负责接收和处理所有外部请求。它将请求分发到不同的处理模块：JupyterClient 处理代码执行相关请求，CommandExecutor 处理Shell命令请求，FileManager 处理文件系统请求，MetricsCollector 处理指标收集请求。这种模块化设计使得每个模块都可以独立发展和优化。

## 五、部署架构图

### 5.1 Docker 本地部署

对于开发测试环境，OpenSandbox 支持直接使用 Docker 进行本地部署。这种部署方式简单快捷，适合个人开发者或小规模测试场景。

在 Docker 部署模式下，OpenSandbox Server 和所有沙箱容器都运行在同一个 Docker 网络中。客户端通过 SDK 与 Server 通信，Server 负责管理所有沙箱的生命周期。沙箱路由器作为反向代理，将客户端对特定沙箱的请求路由到对应的容器端口。

### 5.2 Kubernetes 大规模部署

对于生产环境的大规模应用，OpenSandbox 支持 Kubernetes 部署，能够实现高可用和自动扩缩容。

在 Kubernetes 部署模式下，OpenSandbox Server 以 Pod 的形式运行，通过 Service 实现负载均衡。沙箱实例作为独立的 Pod 由 Kubernetes 进行调度和管理。Ingress Gateway 负责入口流量的路由，Egress Controller 则负责出口流量的控制和安全审计。这种部署模式能够充分利用 Kubernetes 的自动化能力，实现沙箱实例的自动创建、调度和清理。

### 5.3 混合部署架构

在一些复杂的应用场景中，可能需要同时支持本地开发和生产部署。以下架构图展示了一种常见的混合部署模式。

开发者在本地使用 Docker 进行快速迭代和测试，当代码通过验证后，部署到云端的 Kubernetes 集群进行生产运行。容器镜像仓库作为两者之间的桥梁，确保了开发环境和生产环境的一致性。这种混合部署模式既能保证开发效率，又能满足生产环境的稳定性要求。

## 六、网络架构

### 6.1 入口流量路由

OpenSandbox 的入口网关（Ingress Gateway）负责将外部请求路由到正确的沙箱实例。它支持多种路由策略，能够适应不同的应用场景。

入口网关接收外部请求后，首先根据路由规则确定目标沙箱，然后进行认证检查，最后将请求转发到对应的沙箱实例。每个沙箱内部都运行着 execd 守护进程，监听统一的端口（默认 44772），接收来自网关的请求。

### 6.2 出口流量控制

为了确保沙箱的安全性，OpenSandbox 实现了细粒度的出口流量控制（Egress）。这对于防止恶意代码对外发起攻击或泄露敏感数据至关重要。

出口流量控制组件会检查沙箱内部代码的所有对外网络请求，根据预定义的策略决定是否允许访问。所有的出口流量都会被记录到审计日志中，便于安全审计和问题排查。
