# PentAGI 提示词分析文档

## 一、提示词概览

PentAGI 是一个复杂的多代理渗透测试系统，其核心功能依赖于精心设计的提示词工程。本文档分析了PentAGI项目中提示词的设计模式、实现方式和技术细节，帮助理解如何通过大型语言模型驱动自动化渗透测试流程。

## 二、提示词架构

PentAGI采用模块化的提示词设计，每个专业化代理（Agent）都有其对应的提示词模板。系统通过Go语言的文本模板引擎实现动态提示词生成，支持上下文变量插值和条件渲染。

### 2.1 提示词模板位置

项目中的提示词模板存储在以下目录：

```
backend/pkg/templates/prompts/
```

核心提示词模板文件包括：

| 文件名 | 用途 |
|--------|------|
| primary_agent.tmpl | 主编排器代理模板 |
| pentester.tmpl | 渗透测试代理模板 |
| searcher.tmpl | 搜索代理模板 |
| developer.tmpl | 开发代理模板 |
| adviser.tmpl | 顾问代理模板 |
| memorist.tmpl | 记忆代理模板 |
| installer.tmpl | 安装代理模板 |

### 2.2 提示词模板变量

PentAGI的提示词系统使用Go模板语法，通过双大括号包裹变量名实现动态内容插入。主要的模板变量包括：

**执行上下文变量**

| 变量 | 说明 |
|------|------|
| `{{.ExecutionContext}}` | 当前流程、任务和子任务的结构化详细信息（ID、状态、标题、描述） |
| `{{.Lang}}` | 用户偏好的响应语言 |
| `{{.CurrentTime}}` | 执行时间戳 |
| `{{.DockerImage}}` | 代理运行的Docker镜像名称 |
| `{{.Cwd}}` | Docker容器内的默认工作目录 |
| `{{.ContainerPorts}}` | 容器环境中可用/映射的端口 |

**工具名称变量**

PentAGI采用统一的变量命名模式来引用各种工具：

```
# 专家代理调用
{{.SearchToolName}}      # 搜索代理
{{.PentesterToolName}}   # 渗透测试代理
{{.CoderToolName}}       # 编码代理
{{.AdviceToolName}}     # 顾问代理
{{.MemoristToolName}}    # 记忆代理
{{.MaintenanceToolName}} # 维护代理

# 记忆操作
{{.SearchGuideToolName}}    # 检索指南
{{.StoreGuideToolName}}     # 存储指南
{{.SearchAnswerToolName}}   # 检索答案/通用
{{.StoreAnswerToolName}}   # 存储答案/通用
{{.SearchCodeToolName}}    # 检索代码片段
{{.StoreCodeToolName}}     # 存储代码片段

# 结果交付
{{.HackResultToolName}}    # 渗透测试最终报告
{{.SearchResultToolName}}  # 搜索最终报告
{{.FinalyToolName}}        # 编排器子任务完成报告

# 系统与环境工具
{{.SummarizationToolName}} # 摘要工具（仅供系统使用）
{{.TerminalToolName}}      # 终端功能
{{.FileToolName}}         # 文件操作功能
{{.BrowserToolName}}       # 浏览器/抓取功能
```

## 三、提示词设计模式

### 3.1 核心结构

PentAGI的每个提示词模板都遵循统一的层次结构：

```markdown
# [代理专家标题]

[角色定义、主要目标和与PentAGI相关的安全重点]

## 核心能力/知识库
[代理特定技能、与PentAGI任务相关的知识领域]

## 操作环境
<container_constraints>...</container_constraints>

## 命令和工具执行规则
<terminal_protocol>...</terminal_protocol>
<tool_usage_rules>...</tool_usage_rules>

## 记忆系统集成
<memory_protocol>...</memory_protocol>

## 团队协作与委派
<team_specialists>...</team_specialists>
<delegation_rules>...</delegation_rules>

## 摘要感知协议
<summarized_content_handling>...</summarized_content_handling>

## 执行上下文
[解释如何使用 {{.ExecutionContext}} 获取流程/任务/子任务详情]

## 完成要求
[输出格式、最终工具使用、语言、报告需求]

{{.ToolPlaceholder}}
```

### 3.2 XML定界符使用

PentAGI使用语义化的XML标签来组织复杂指令，这种设计有以下优势：

- **逻辑分组**：相关指令被组织在独立的XML标签内，便于LLM理解和执行
- **一致性**：所有代理提示词共享概念的XML结构保持一致，确保可预测性
- **可维护性**：修改特定协议只需更新对应的XML标签内容

关键XML标签包括：

| 标签 | 用途 |
|------|------|
| `<container_constraints>` | Docker容器运行环境的限制 |
| `<terminal_protocol>` | 终端命令执行协议 |
| `<tool_usage_rules>` | 工具使用规则 |
| `<memory_protocol>` | 记忆系统交互协议 |
| `<team_specialists>` | 可用的专家代理列表 |
| `<delegation_rules>` | 委派规则和标准 |
| `<summarized_content_handling>` | 摘要内容处理方式 |
| `<search_tools>` | 搜索工具优先级矩阵 |

## 四、关键提示词组件分析

### 4.1 记忆系统协议

记忆系统是PentAGI的核心创新之一，提示词中明确指定了记忆交互规则：

```xml
<memory_protocol>
- 代理必须首先尝试从记忆中使用检索工具（如 {{.SearchGuideToolName}}、{{.SearchAnswerToolName}}）
  检索相关信息，再执行外部操作如网页搜索或运行发现工具
- 仅存储有价值的、新颖的、可重用的知识（如确认的漏洞、成功复杂的命令序列、
  有效的故障排除步骤、可重用的代码片段）
- 避免用琐碎或中间结果污染记忆
</memory_protocol>
```

**设计要点**：

- 强调检索优先原则：先查记忆，再做外部搜索
- 明确存储标准：只存有价值的信息
- 指定具体工具名称：通过模板变量引用

### 4.2 终端执行协议

PentAGI的提示词包含严格的终端命令执行规则：

```xml
<terminal_protocol>
- 命令在隔离的Docker容器中执行，工作目录（cwd）在工具调用之间不持久
- 如需特定路径上下文，必须在单个工具调用中明确执行目录变更
- 对文件操作使用绝对路径
- 为可能长时间运行的命令指定超时处理和输出重定向
- 限制重复相同失败命令的次数（最多3次尝试）
- 鼓励使用非交互标志避免挂起
</terminal_protocol>
```

**设计要点**：

- 明确执行环境的特殊性（容器隔离、非持久工作目录）
- 防止路径相关错误（强制绝对路径）
- 避免命令挂起（超时、非交互标志）
- 限制重试次数，避免无效循环

### 4.3 团队协作协议

PentAGI采用多代理架构，提示词中定义了完整的专家代理协作机制：

```xml
<team_specialists>
- searcher: 信息检索专家
- pentester: 渗透测试专家
- developer: 开发专家
- adviser: 顾问专家
- memorist: 记忆管理专家
- installer: 安装维护专家

<delegation_rules>
- 先尝试使用自己的工具/知识独立解决
- 仅当任务明确超出核心技能或专家代理明显更好时才委派
- 每次委派请求必须提供完整上下文，包括背景信息、具体目标、已收集的数据/发现、约束条件
</delegation_rules>
```

### 4.4 摘要感知协议

由于PentAGI使用上下文摘要来管理令牌限制，提示词包含专门的处理指南：

```xml
<summarized_content_handling>
- 系统生成两种形式的摘要：工具调用摘要和前缀摘要
- 代理应将摘要严格视为实际过去事件的历史记录
- 从摘要中提取有用信息（过去的命令、成功、失败、错误、发现）以指导当前策略
- 严格禁止：模仿摘要格式、使用摘要前缀、调用摘要工具
- 系统仅通过结构化工具调用运行，任何模拟操作或结果的纯文本尝试都将失败
</summarized_content_handling>
```

## 五、代理特定提示词

### 5.1 编排器代理（Primary Agent）

编排器是PentAGI的核心协调者，负责任务分解和代理委派：

| 关注点 | 说明 |
|--------|------|
| 核心职责 | 任务分解、委派编排、跨子任务的上下文管理、最终子任务结果聚合 |
| 关键指令 | 委派前收集上下文、严格强制当前子任务范围、委派时提供完整上下文、管理执行尝试/失败、使用{{.FinalyToolName}}报告子任务完成状态和综合结果 |
| 完成要求 | 使用{{.FinalyToolName}}交付最终报告 |

### 5.2 渗透测试代理（Pentester）

渗透测试代理执行实际的安全测试操作：

| 关注点 | 说明 |
|--------|------|
| 核心职责 | 实际安全测试、工具执行（nmap、sqlmap等）、漏洞利用、证据收集和文档化 |
| 关键指令 | 首先检查记忆、严格遵守终端规则和容器约束、仅使用列出的可用工具、适当委派、提供详细的证据支持的利用报告 |
| 完成要求 | 使用{{.HackResultToolName}}交付最终报告 |

### 5.3 搜索代理（Searcher）

搜索代理专注于高效信息检索：

| 关注点 | 说明 |
|--------|------|
| 核心职责 | 高效信息检索（内部记忆和外部源）、源评估和优先级排序、发现综合 |
| 关键指令 | 始终优先记忆搜索、严格限制搜索操作数量、为查询复杂度使用正确工具、获得足够信息后立即停止搜索、通过{{.SearchResultToolName}}交付简洁但全面的综合结果 |
| 完成要求 | 使用{{.SearchResultToolName}}交付最终报告 |

## 六、提示词设计原则

### 6.1 核心设计原则

**结构与组织**

- 使用Markdown标题实现清晰的视觉层次
- 从高级角色定义到具体协议和要求，逻辑流程清晰
- 将关键操作约束（安全、环境）放在提示词开头以提高可见性
- 使用单独、明确标记的部分

**语义化XML定界符**

- 使用描述性XML标签逻辑分组相关指令
- 保持跨提示词的标签命名和结构一致性
- 适当使用嵌套定义

**上下文窗口优化**

- 根据重要性优先排序信息
- 使用压缩技术处理冗长信息
- 对复杂提示词进行逻辑分组

### 6.2 代理特定定制

**基于角色的定制**

- 根据代理在PentAGI中的专业化角色定制指令、语气、知识引用和复杂度
- 对具有直接系统/工具访问权限的代理执行更严格的命令协议和安全措施
- 包含对代理功能相关的专门知识库或工具集的引用

**安全与操作边界**

- 明确说明允许操作的范围和安全约束
- 在`<container_constraints>`中定义Docker容器限制
- 明确禁止操作，使用大写强调关键安全警告

### 6.3 代理能力与持久性

**代理持久性协议**

```markdown
你是一个代理——继续工作直到子任务完全完成。
不要过早结束你的回合或将控制权交还给用户/编排器，
直到你实现了当前子任务的具体目标。
```

**规划与推理**

- 要求代理在行动前明确规划
- 鼓励链式思考推理
- 对关键安全任务强制验证步骤

**错误处理**

- 提供明确的意外错误处理指导
- 定义最大重试阈值（通常3次）
- 包括优雅降级的说明

## 七、提示词优化建议

### 7.1 有效模式

| 模式 | 说明 |
|------|------|
| 渐进式披露 | 按复杂性递增的层次引入概念 |
| 明确排序 | 对顺序操作使用编号步骤或清晰序列标记 |
| 任务分解 | 将复杂任务分解为具有各自指南的明确定义的子任务 |
| 参数验证 | 包含在继续操作之前验证输入的说明 |
| 后备链 | 定义主要方法失败时的明确替代方案 |

### 7.2 常见反模式

| 反模式 | 说明 |
|--------|------|
| 过度规定 | 提供过多约束导致决策瘫痪 |
| 冲突优先级 | 在没有明确层次的情况下给出矛盾指导 |
| 模糊成功标准 | 未定义任务何时被视为完成 |
| 隐含假设 | 依赖未声明的知识或上下文 |
| 工具模糊 | 对特定情况使用哪些工具缺乏清晰指导 |

### 7.3 改进建议

基于对PentAGI提示词工程的深入分析，以下是一些潜在的优化方向：

**记忆系统优化**

- 当前记忆协议主要关注检索和存储时机，建议增加：
  - 记忆更新策略（何时更新已有记忆）
  - 记忆冲突解决机制
  - 记忆质量评估标准

**安全边界增强**

- 当前提示词已包含基本安全约束，可进一步增强：
  - 添加对新兴攻击向量的动态响应指导
  - 引入更细粒度的权限分级机制
  - 增加对第三方工具的安全审查提示

**多代理协作优化**

- 建议增加：
  - 代理间通信效率优化
  - 冲突检测和解决机制
  - 协作结果的验证流程

**上下文管理**

- 当前的摘要协议是核心亮点，建议增强：
  - 动态摘要触发阈值配置
  - 摘要质量评估标准
  - 关键信息保留策略

## 八、总结

PentAGI的提示词工程体现了以下关键设计理念：

1. **模块化设计**：每个代理有独立提示词模板，便于维护和扩展
2. **结构化协议**：使用XML标签实现复杂指令的清晰组织
3. **上下文感知**：通过模板变量实现动态内容生成
4. **记忆优先**：强调先检索记忆再做外部操作的原则
5. **多代理协作**：清晰的委派规则和专家代理定义
6. **安全第一**：严格的容器约束和操作边界定义

这种提示词架构使PentAGI能够有效地协调多个专业化AI代理，完成复杂的自动化渗透测试任务。
