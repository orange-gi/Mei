# Karpathy 启发式 Claude Code 指南：Why-What-How 深度解析

## Why（为什么）：解决 LLM 编程的根因问题

### 背景与动机

在 AI 辅助编程日益普及的今天，LLM（如 Claude、GPT 等）在代码生成方面展现出惊人的能力。然而，深度学习先驱 Andrej Karpathy 在社交媒体上指出了一个关键问题：**LLM 在编程时存在系统性缺陷**，这些问题不是偶发的 bug，而是根植于模型行为的模式性问题。

Karpathy 观察到，LLM 在编程时倾向于：

- **静默假设**：在用户意图不明确时，模型会替用户做出假设并执行，而不是主动询问
- **隐藏困惑**：当遇到不明确的情况时，模型不会表达困惑或寻求澄清，而是盲目猜测
- **不暴露权衡**：在存在多种解决方案时，模型不会呈现利弊分析，而是直接选择一个
- **过度复杂化**：代码和 API 往往被过度设计，添加了大量抽象层次，而实际上简单的实现就足够
- **不恰当地修改**：有时候会修改或删除那些理解不够深入的代码和注释，即使这些修改与任务正交（正交意味着不应该被影响）

### 问题的本质

这些问题并非源于技术实现，而在于 LLM 的**行为模式**。LLM 被训练为"尽量完成用户请求"，这种默认的服从性导致它们：

1. **过度补充**：假设用户需要比实际请求更多的功能
2. **过早优化**：添加"灵活性"和"可配置性"来应对假想的未来需求
3. **过度工程化**：应用设计模式和最佳实践，即使简单的代码就能解决问题

### 目标用户

本项目面向以下用户群体：

- **开发者**：使用 Claude Code 等 AI 编程助手的程序员
- **团队负责人**：希望规范化 AI 辅助编程流程的技术领导
- **AI 工程师**：研究 AI 编程行为的从业者
- **工具开发者**：构建 AI 编程工具的团队

## What（是什么）：四个核心原则的系统化实践

### 项目概述

**Karpathy-Inspired Claude Code Guidelines** 是一个简单而强大的项目，旨在通过四个核心原则改善 LLM 的编程行为。这四个原则直接针对 Andrej Karpathy 提出的问题：

| 原则 | 针对问题 |
|------|----------|
| **Think Before Coding** | 错误假设、隐藏困惑、缺失权衡 |
| **Simplicity First** | 过度复杂化、臃肿抽象 |
| **Surgical Changes** | 正交编辑、触碰不应修改的代码 |
| **Goal-Driven Execution** | 循环直到满足特定目标 |

### 核心功能

#### 1. Think Before Coding（三思而后行）

**核心理念**：不要假设，不要隐藏困惑，主动呈现权衡。

具体要求：

- **显式陈述假设**：如果不确定，主动询问而不是猜测
- **呈现多种解释**：当存在歧义时，不要静默选择，而是呈现所有可能的理解
- **在必要时反驳**：如果存在更简单的方案，主动提出
- **困惑时停止**：命名不明确之处，主动寻求澄清

#### 2. Simplicity First（简单优先）

**核心理念**：最小代码解决当前问题，不做投机性设计。

具体要求：

- 不添加未请求的功能
- 不为单次使用的代码创建抽象
- 不添加"灵活性"或"可配置性"
- 不处理不可能的场景
- 如果 200 行可以变成 50 行，重写

**自测问题**：一位资深工程师会说这太复杂了吗？如果是，简化。

#### 3. Surgical Changes（精准修改）

**核心理念**：只触碰必须修改的部分，只清理自己造成的混乱。

具体要求：

- 不要"改进"相邻的代码、注释或格式
- 不要重构没有损坏的部分
- 匹配现有风格，即使你会有不同的做法
- 如果发现无关的死代码，提及而不是删除

当修改造成孤立代码时：

- 移除因本次修改而变成未使用的导入/变量/函数
- 不要移除预先存在的死代码，除非被要求

**自测**：每行修改的代码都应该能追溯到用户的请求。

#### 4. Goal-Driven Execution（目标驱动执行）

**核心理念**：定义成功标准，循环直到验证通过。

具体要求：

将命令式任务转化为可验证的目标：

| 不是... | 转化为... |
|---------|----------|
| "添加验证" | "为无效输入编写测试，然后使其通过" |
| "修复 bug" | "编写测试复现它，然后使其通过" |
| "重构 X" | "确保测试在重构前后都通过" |

对于多步骤任务，陈述简要计划：

```
1. [步骤] → 验证：[检查]
2. [步骤] → 验证：[检查]
3. [步骤] → 验证：[检查]
```

强成功标准让 LLM 可以独立循环。弱标准（"让它工作"）需要不断澄清。

## How（怎么做）：快速上手与最佳实践

### 快速开始

#### 方式一：Claude Code 插件（推荐）

在 Claude Code 中添加市场并安装插件：

```bash
# 添加市场
/plugin marketplace add forrestchang/andrej-karpathy-skills

# 安装插件
/plugin install andrej-karpathy-skills@karpathy-skills
```

这将把指南作为 Claude Code 插件安装，使其在所有项目中可用。

#### 方式二：CLAUDE.md（按项目）

**新项目**：

```bash
curl -o CLAUDE.md https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md
```

**现有项目（追加）**：

```bash
echo "" >> CLAUDE.md
curl https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md >> CLAUDE.md
```

### 核心使用方式

#### 1. 融合项目特定指南

这些指南设计为可与项目特定指令合并使用。添加到现有 CLAUDE.md 或创建新文件：

```markdown
## 项目特定指南

- 使用 TypeScript 严格模式
- 所有 API 端点必须有测试
- 遵循 `src/utils/errors.ts` 中现有的错误处理模式
```

#### 2. 判断指南是否生效

使用这些指南后，你应该观察到：

- **Diff 中不必要的更改减少**：只出现请求的更改
- **因过度复杂化导致的重写减少**：代码第一次就简单
- **澄清问题出现在实现之前**：而不是在错误之后
- **PR 简洁干净**：没有顺便的重构或"改进"

#### 3. 权衡说明

这些指南偏向**谨慎而非速度**。对于琐碎任务（简单的拼写错误修复、明显的单行代码），使用判断力——不是每个更改都需要全套流程。

目标是减少非平凡工作中的高成本错误，而不是减慢简单任务的速度。

### 最佳实践建议

#### 1. 从简单开始

```python
# 过度工程化
def calculate_discount(amount: float, percent: float) -> float:
    # 100行策略模式代码...

# 简单优先
def calculate_discount(amount: float, percent: float) -> float:
    return amount * (percent / 100)
```

#### 2. 在修改前询问

```
用户：添加用户数据导出功能

LLM 应该问：
1. 导出所有用户还是过滤子集？（隐私影响）
2. 您说的"导出"是指：
   - 浏览器下载文件？
   - 后台任务发送邮件？
   - API 端点返回数据？
3. 包含哪些用户字段？（某些可能敏感）
4. 通常有多少用户？（影响方法选择）
```

#### 3. 定义可验证的目标

```
用户：让搜索更快

LLM 应该呈现：
"让搜索更快"可能有几种含义：

1. 更快的响应时间（< 100ms 而不是 500ms）
   - 添加数据库索引
   - 缓存频繁查询
   - 预计工作量：2小时

2. 处理更多并发搜索（吞吐量）
   - 添加异步/等待
   - 连接池
   - 预计工作量：4小时

3. 更快的感知速度（用户体验）
   - 立即显示部分结果
   - 渐进式加载
   - 预计工作量：3小时

当前搜索典型查询耗时约500ms。
您最关心哪方面？
```

#### 4. 精准修改

```diff
# 只修改必要的行
  def validate_user(user_data):
-     if not user_data.get('email'):
+     email = user_data.get('email', '')
+     if not email or not email.strip():
          raise ValueError("Email required")
```

## 关键洞察

Andrej Karpathy 的核心观点是：

> "LLM 在循环直到满足特定目标方面表现出色... 不要告诉它要做什么，给它成功标准，然后看它行动。"

**Goal-Driven Execution** 原则正是这一洞察的具体体现：将命令式指令转化为带有验证循环的声明性目标。

这种方法的成功案例包括：

- **TDD（测试驱动开发）**：定义测试作为成功标准
- **验收标准**：用户故事中的可验证条件
- **SLO/SLI**：服务级别目标和指标

## 总结

**Karpathy-Inspired Claude Code Guidelines** 是一个小巧但影响深远的项目。它通过四个简单但强大的原则，直接解决了 LLM 在编程时遇到的实际问题：

1. **主动沟通**：不隐藏假设和困惑
2. **保持简单**：抵制过度工程的冲动
3. **精准操作**：只触碰必要的部分
4. **目标验证**：定义成功标准并循环验证

这个项目证明了，有时候**减少 AI 的行为**比增强它更能提高输出质量。**好的代码是今天解决今天问题的简单代码，而不是过早解决明天问题的复杂代码。**
