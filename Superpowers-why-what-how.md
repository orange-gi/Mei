# Superpowers 项目介绍

> 基于Why-What-How框架的系统性分析

---

## 一、Why（为什么）—— 项目背景与动机

### 1.1 解决的核心问题

Superpowers 旨在解决**AI编码代理在软件开发过程中的系统性缺陷**。当前AI代理在代码生成场景中面临以下关键挑战：

| 问题类型 | 具体表现 | 造成的影响 |
|---------|---------|-----------|
| **需求理解偏差** | 直接跳入编码，忽略用户真实意图 | 构建错误的产品，浪费开发资源 |
| **计划缺失** | 缺乏系统性实施计划 | 代码质量参差不齐，项目延期 |
| **测试缺失** | 不遵循测试驱动开发原则 | 代码可靠性低，bug频发 |
| **审查不足** | 缺少有效的代码审查机制 | 技术债务累积，代码库恶化 |
| **上下文丢失** | 长时间任务中偏离原始目标 | 任务迷失方向，产出质量下降 |

### 1.2 项目的起源与背景

Superpowers 由 Jesse Vincent（@obra）创建并维护，源于对AI编码代理工作方式的深刻洞察：

- **观测发现**：AI代理在处理复杂软件开发任务时，往往"想到哪写到哪"，缺乏结构化的工程实践
- **行业需求**：随着Claude Code、Codex、OpenCode等AI编程工具的普及，亟需一套规范化的开发框架
- **核心洞见**：AI代理需要的不只是代码生成能力，更需要**工程化的思维模式和标准化的开发流程**

### 1.3 目标用户群体

Superpowers 主要面向以下用户群体：

1. **AI编码代理平台**（Claude Code、Codex、OpenCode等）
2. **使用AI辅助开发的软件工程师**
3. **需要规范化AI代码生成的研发团队**
4. **构建AI编程工具的开发者**

### 1.4 为什么现有方案不够好

传统AI编码解决方案存在以下局限性：

| 现有方案 | 局限性 |
|---------|--------|
| 通用LLM直接生成代码 | 缺乏工程化流程，无测试驱动，无审查机制 |
| 简单的代码生成提示词 | 过于碎片化，不成体系 |
| 单一功能的开发工具 | 无法覆盖完整开发周期 |
| 纯人工审查流程 | 无法规模化，难以与AI代理集成 |

Superpowers 的独特价值在于提供**完整、可组合、可演进**的技能体系，涵盖从需求澄清到代码审查的全流程。

## 二、What（是什么）—— 项目核心定义

### 2.1 项目概述

Superpowers 是**一套为AI编码代理设计的完整软件开发工作流框架**，它通过可组合的"技能"（Skills）和标准化指令，确保AI代理能够遵循专业软件工程实践进行开发工作。

```
┌─────────────────────────────────────────────────────────────┐
│                    Superpowers 核心定位                       │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐          │
│  │   技能库    │  │   工作流程  │  │   最佳实践  │          │
│  │ (Skills)    │  │ (Workflows) │  │ (Best Practices)│    │
│  └─────────────┘  └─────────────┘  └─────────────┘          │
│         │              │                 │                   │
│         └──────────────┼─────────────────┘                   │
│                        ▼                                     │
│              ┌─────────────────────┐                         │
│              │  AI编码代理工作流框架 │                        │
│              └─────────────────────┘                         │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 核心功能模块

Superpowers 包含四大核心功能模块：

#### 2.2.1 技能库（Skills）

技能是Superpowers的基本构建块，每个技能封装了特定的专业能力：

```
技能分类
├── 测试技能
│   └── test-driven-development - 红色-绿色-重构循环
├── 调试技能
│   ├── systematic-debugging - 四阶段根本原因分析
│   └── verification-before-completion - 完成前验证
├── 协作技能
│   ├── brainstorming - 苏格拉底式设计精炼
│   ├── writing-plans - 详细实施计划
│   ├── executing-plans - 批量执行与检查点
│   ├── dispatching-parallel-agents - 并发子代理工作流
│   ├── requesting-code-review - 预审检查清单
│   ├── receiving-code-review - 响应反馈
│   ├── using-git-worktrees - 并行开发分支
│   ├── finishing-a-development-branch - 合并/PR决策
│   └── subagent-driven-development - 两阶段审查迭代
└── 元技能
    ├── writing-skills - 创建新技能
    └── using-superpowers - 技能系统介绍
```

#### 2.2.2 命令系统（Commands）

提供CLI命令接口，实现标准化的开发操作：

| 命令 | 功能描述 |
|-----|---------|
| `/superpowers:brainstorm` | 交互式设计精炼会话 |
| `/superpowers:write-plan` | 创建结构化实施计划 |
| `/superpowers:execute-plan` | 批量执行计划任务 |

#### 2.2.3 钩子系统（Hooks）

自动化技能触发机制，实现智能化的开发辅助：

- 根据上下文自动推荐相关技能
- 在关键节点触发相应的质量检查
- 确保开发流程的规范执行

#### 2.2.4 代理系统（Agents）

子代理的创建、调度和管理框架：

- 任务分派与管理
- 并发执行支持
- 状态追踪与恢复

### 2.3 主要特性

Superpowers 具有以下显著特性：

1. **模块化设计**：每个技能独立可复用，可根据需求自由组合
2. **渐进式执行**：从简单任务到复杂项目，均可适用
3. **质量优先**：强制执行TDD、代码审查等质量保证实践
4. **上下文感知**：智能识别当前开发阶段，推荐相应技能
5. **多平台支持**：兼容Claude Code、Codex、OpenCode等主流平台

### 2.4 技术栈概述

Superpowers 采用多语言混合开发：

| 语言 | 占比 | 用途 |
|-----|-----|------|
| Shell | 67.9% | 脚本、钩子、系统集成 |
| JavaScript | 22.8% | 交互逻辑、工具函数 |
| Python | 5.1% | 辅助工具、脚本 |
| TypeScript | 3.8% | 类型安全的组件开发 |
| Batchfile | 0.4% | 特定平台脚本 |

### 2.5 与竞品的差异化

| 维度 | Superpowers | 通用AI编程工具 |
|-----|-------------|---------------|
| **方法论** | 系统化工程实践 | 自由编码 |
| **质量保证** | TDD强制执行 | 无强制要求 |
| **流程覆盖** | 全生命周期 | 仅代码生成 |
| **可组合性** | 技能可自由组合 | 单一工具链 |
| **可扩展性** | 开放的技能框架 | 封闭系统 |

### 3.2 核心使用方式

#### 3.2.1 基本工作流程

Superpowers 定义了标准化的六阶段开发流程：

```
┌──────────────────────────────────────────────────────────────────┐
│                    Superpowers 标准开发流程                        │
└──────────────────────────────────────────────────────────────────┘

  ┌──────────┐
  │  Phase 1 │  brainstorming（头脑风暴）
  │          │  ─────────────────────────
  │          │  在编写任何代码前，澄清需求
  │          │  精炼设计思路，探索替代方案
  └────┬─────┘
       │
       ▼
  ┌──────────┐
  │  Phase 2 │  using-git-worktrees（创建工作空间）
  │          │  ───────────────────────────────
  │          │  建立隔离的开发环境
  │          │  准备干净的测试基线
  └────┬─────┘
       │
       ▼
  ┌──────────┐
  │  Phase 3 │  writing-plans（编写计划）
  │          │  ─────────────────────────────
  │          │  将工作分解为小块任务
  │          │  每个任务2-5分钟可完成
  └────┬─────┘
       │
       ▼
  ┌──────────┐
  │  Phase 4 │  subagent-driven-development（子代理开发）
  │          │  ──────────────────────────────────────
  │          │  派生子代理执行任务
  │          │  两阶段代码审查
  └────┬─────┘
       │
       ▼
  ┌──────────┐
  │  Phase 5 │  test-driven-development（测试驱动开发）
  │          │  ───────────────────────────────────────
  │          │  红色-绿色-重构循环
  │          │  贯穿整个实施过程
  └────┬─────┘
       │
       ▼
  ┌──────────┐
  │  Phase 6 │  finishing-a-development-branch（完成分支）
  │          │  ────────────────────────────────────────
  │          │  验证测试通过
  │          │  合并或创建PR
  │          │  清理工作树
  └──────────┘
```

#### 3.2.2 阶段详解

**阶段1：Brainstorming（头脑风暴）**

```markdown
使用时机：任何创造性工作之前

核心原则：
1. 先了解项目状态（查看文件、文档、最近提交）
2. 每次只问一个问题
3. 优先使用选择题而非开放性问题
4. 探索2-3种不同方案并比较优劣
5. 以200-300字为单位展示设计，逐一验证

输出：设计文档（docs/plans/YYYY-MM-DD--design.md）
```

**阶段2：Using Git Worktrees（创建工作空间）**

```markdown
使用时机：设计方案批准后

操作步骤：
1. 为新功能创建独立的git worktree
2. 在新分支上工作
3. 运行项目初始化/设置
4. 验证干净的测试基线

优势：隔离开发环境，互不干扰
```

**阶段3：Writing Plans（编写计划）**

```markdown
使用时机：开始实现前

任务分解原则：
- 每个任务2-5分钟可完成
- 每个任务包含：
  * 精确的文件路径
  * 完整的实现代码
  * 验证步骤

输出：结构化的任务列表
```

**阶段4：Subagent-Driven Development（子代理驱动开发）**

```markdown
使用时机：大规模实施阶段

工作模式：
1. 为每个任务派遣新子代理
2. 两阶段审查：
   - 规范合规性审查
   - 代码质量审查
3. 子代理自主工作数小时

关键优势：
- 并行处理多个任务
- 保持专注，不偏离计划
```

**阶段5：Test-Driven Development（测试驱动开发）**

```markdown
使用时机：实施过程中

TDD循环：
1. 红色（Red）：编写失败的测试
2. 绿色（Green）：编写最少代码使测试通过
3. 重构（Refactor）：优化代码，保持测试通过

严格规则：永远先写测试
```

**阶段6：Finishing a Development Branch（完成分支）**

```markdown
使用时机：所有任务完成后

检查清单：
□ 所有测试通过
□ 代码审查完成
□ 文档已更新

决策选项：
- 合并到主分支
- 创建Pull Request
- 保留分支继续工作
- 丢弃分支
```

### 3.4 最佳实践建议

#### 3.4.1 技能使用原则

```
✅ 推荐做法：
- 遇到任何开发任务，先检查是否有适用技能
- 即使只有1%可能性适用，也使用技能
- 完全按照技能描述执行，不擅自偏离
- 技能有清单时，使用TodoWrite逐项执行

❌ 禁止做法：
- 认为"简单任务"就不检查技能
- 先行动再检查技能
- 跳过技能检查直接响应
- 使用Read工具读取技能文件（必须用Skill工具）
```

#### 3.4.2 技能优先级

当多个技能可能同时适用时，按以下顺序使用：

```
优先级顺序：
1. 流程技能（brainstorming, debugging）
   → 决定如何处理任务

2. 实现技能（frontend-design, mcp-builder）
   → 指导具体执行

示例：
- "构建功能X" → brainstorming → 实现技能
- "修复bug" → debugging → 领域特定技能
```

#### 3.4.3 创建新技能

如需扩展技能库，遵循以下流程：

```markdown
1. Fork仓库
2. 创建新分支
3. 参照writing-skills技能模板创建技能
4. 使用subagent测试新技能
5. 提交Pull Request

技能模板要求：
- SKILL.md：完整的技能定义
- 清晰的参数说明
- 详细的使用场景
- 完整的示例
```

### 3.5 核心哲学

Superpowers 遵循四大核心原则：

| 原则 | 含义 | 实践体现 |
|-----|-----|---------|
| **测试驱动开发** | 始终先写测试 | RED-GREEN-REFACTOR循环 |
| **系统化方法** | 流程优于猜测 | 标准化工作流 |
| **复杂性降低** | 简单是首要目标 | YAGNI原则 |
| **证据而非断言** | 验证后再宣布成功 | 强制测试验证 |
